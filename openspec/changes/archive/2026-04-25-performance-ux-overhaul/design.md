## Context

当前系统使用 FastAPI 作为后端，SQLAlchemy 处理数据库，腾讯云 COS 存储图片。生图流程通过 `BackgroundTasks` 异步处理，但由于 COS SDK 是同步的且在任务中持有数据库连接，导致了性能瓶颈。前端画廊直接加载原图，缺乏针对移动端和弱网环境的优化。

## Goals / Non-Goals

**Goals:**
- 实现全链路非阻塞的生图与转存逻辑。
- 通过 UUID 混淆和公有读保障分享体验与基本的隐私安全。
- 实现“积分冻结”两阶段扣费模式，提升用户体验并防范并发漏洞。
- 利用 COS 数据万象实现画廊“占位模糊 -> 缩略图 -> 原图”的平滑加载。
- 在“我的创作”中实现 `pending` 状态的占位卡片渲染。

**Non-Goals:**
- 引入 Celery/Redis 等复杂的任务队列。
- 实现 Prompt 自动扩充功能（留待二期）。

## Decisions

### 1. 采用 UUID 混淆公有链接
- **方案**: COS 保持公有读，文件名采用 `uuid.uuid4()`。
- **理由**: 支持用户直接复制图片链接分享到社交媒体，且无需后端频繁计算签名，性能最优。

### 2. 列表页“实时占位”与 LQIP 加载
- **方案**: 
    - 对 `pending` 状态的记录，前端展示为毛玻璃效果的占位卡片，并显示对应 Prompt。
    - 对 `success` 图片，利用 COS 数据万象拼接缩略图参数（如 `?imageMogr2/thumbnail/400x/quality/60/interlace/1`），实现“模糊预览 -> 高清加载”的平滑过渡。
- **理由**: 提供“确定性反馈”，消除用户在长时异步任务中的焦虑感。

### 3. 计费系统：两阶段冻结模式 (方案 B)
- **方案**: 在 `User` 表增加 `frozen_points` 字段。提交任务时将对应积分从 `points` 转移至 `frozen_points`；任务终态为 `success` 时扣除冻结额度，终态为 `failed` 时回退至 `points` 并销账冻结额度。
- **理由**: 解决用户在生图期间积分被“瞬间扣空”的心理落差，同时有效防止并发扣费漏洞。

### 4. 数据库连接“按需获取”
- **方案**: 在后台任务 `process_image_task` 中，将任务分为三个阶段：初始化（查库）、AI 生成（长时、无库操作）、结果更新（查库更新）。在 AI 生成阶段前关闭 session，完成后重新获取。
- **理由**: 释放数据库连接池，避免在大并发生图时连接被长时间挂起。

## Risks / Trade-offs

- **[Risk] 积分冻结状态丢失** → **[Mitigation]** 如果服务器在生图期间意外宕机，重启后需有清理脚本将 `frozen_points` 归还给 `points`，并标记任务为 `failed`。
- **[Risk] 用户在历史页长时间停留无法看到更新** → **[Mitigation]** 前端对 `pending` 状态的任务进行短频次轮询（如 5 秒一次），直到变为成功。
