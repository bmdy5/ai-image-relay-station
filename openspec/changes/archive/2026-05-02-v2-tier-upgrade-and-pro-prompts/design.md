## Context

当前生图系统通过 `backend/api/image.py` 处理生成逻辑，前端 `HomePage.jsx` 仅提供简单的输入框和三个固定的质量选项。提示词处理非常原始，缺乏针对不同艺术风格的自动化增强。

## Goals / Non-Goals

**Goals:**
- 实现 5/10/15 分级定价逻辑。
- 建立后端的提示词模版库，支持动态占位符替换。
- 在前端实现风格与输入框的联动（预填提示词）。
- 开启“图生图”功能接口，仅限中、高级用户。

**Non-Goals:**
- 不引入真正的“超分辨率”放大 API（仅通过提示词模拟）。
- 不开发实时的“大师笔记” AI 写作功能（暂定静态提示词）。

## Decisions

### 1. 提示词包装策略 (Prompt Wrapping Strategy)
**决定**：在后端实现 `wrap_prompt` 逻辑，而不是在前端。
**理由**：保护核心 Prompt 资产，且方便未来进行 A/B 测试或后端调整模版。
**替代方案**：前端处理。缺点是模版容易泄露且更新不便。

### 2. 交互引导逻辑
**决定**：风格切换时，如果输入框为空或仅包含上一个风格的模版，则强制覆盖。若有用户自定义内容，则弹窗确认。
**理由**：平衡“自动引导”与“保护用户劳动成果”。

### 3. 图生图门控与后端透传
**决定**：前端增加 `requiresImage` 标识。后端检测到 `quality` 为 `hd` 或 `master` 且携带图片 URL 时，调用 AI 接口的图片参考参数。
**逻辑**：中级和高级用户上传图片后，提示词将作为图片内容的“指令”而非仅仅是描述。

### 4. 极端画质增强 (Ultra HD Injection)
**决定**：在 `master` 档位下，后端强制在最终提示词末尾追加特定的高画质后缀。
**关键词**：`8k, hyper-detailed, masterpiece, professional cinematic lighting, sharp focus`。

## Risks / Trade-offs

- [风险] 提示词模版过长导致 API 报错 → [缓解] 限制用户输入的前缀长度在 1000 字以内，确保总 Prompt 不超过模型上限。
- [风险] 用户删除占位符导致模版失效 → [缓解] 采用兜底策略，若匹配不到占位符，则将用户输入作为全局主题嵌入。
- [风险] 5/10/15 计费逻辑变更可能导致旧版本前端产生冲突 → [缓解] 后端接口需向下兼容，若未传新参数则按旧标准 5 积分执行。
