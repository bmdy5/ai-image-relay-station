## 1. 计费核心规则（代码实测）

### 1.1 扣费时序（激进型）
1. **预扣**：`POST /image/generate` 发起时，立即将 `cost` 写入 `frozen_points`，检查 `points - frozen_points >= cost`，不足则拒绝。
2. **成功扣费**：任务成功后，从 `points` 中真正扣除，同时释放 `frozen_points`。
3. **失败处理**：仅释放 `frozen_points`，**不退还积分**（因为 `points` 从未被扣过）。

> 注意：当前代码逻辑是失败不扣费（因为预扣是写 `frozen_points`，`points` 只在成功后才变动）。这与"激进策略"存在差异，若要实现"发起即扣费"，需修改 `process_image_task` 中的扣费节点。

### 1.2 三档定价

| 档位 | 积分 | API 尺寸 | 成本估算（RMB） | 毛利空间 |
| :--- | :--- | :--- | :--- | :--- |

## 3. 邀请返利体系（已上线，代码实测）

| 触发条件 | 获益方 | 积分 | 流水前缀 |
| :--- | :--- | :--- | :--- |
| 受邀者注册时填写邀请码 | 受邀者 | +5 积分 | `INVITE_JOIN_*` |
| 受邀者完成**首次**成功生图 | 邀请人 | +10 积分 | `INVITE_REWARD_*` |

**防薅羊毛限制**：
- 邀请人每日最多因此获得 **5 次** 奖励（`can_receive_invitation_reward` 函数控制）。
- 同一受邀者只能为邀请人触发一次奖励（`is_invitee_rewarded` 函数控制）。

## 4. 两套充值入口（并存）

### 4.1 在线支付（主）
通过 `POST /payment/create` → 跳转支付乐 → 回调 `GET /payment/notify` → 自动到账。
管理员**不能手动干预**在线支付订单（代码强制校验 `out_trade_no is None`）。

### 4.2 人工报备（辅）
用户上传截图，通过 `POST /user/recharge/apply` 提交。
管理员在后台审核，通过则积分自动到账，驳回则更新状态为 `rejected`。
限制：**同一用户同时只能有 1 条 `pending` 报备记录**。

## 5. 管理后台能力

- **仪表盘**：实时显示总用户数、总收入（元）、总积分消耗、风格使用排行、最近 20 条全站动态。
- **违规处理**：`DELETE /admin/image/{log_id}/wipe` 同步删除数据库记录和 COS 文件。
- **邀请审计**：`GET /admin/invitation/logs` 查看所有邀请奖励流水，含 IP 和指纹信息（用于核查薅羊毛行为）。
